# 日常迭代工作流

这一篇面向已经进入持续开发阶段的使用者。

## 适合什么场景

- 每天都在改同一个工程
- 页面、资源、代码都在一起迭代
- 需要频繁保存、预览、回看历史

## 推荐的日常节奏

一个比较稳的循环通常是：

1. 打开 Recent Project
2. 先看 Diagnostics 有没有遗留错误
3. 做一小批页面修改
4. 保存
5. 做一次预览验证
6. 记录或处理资源变化

## 页面改动的建议粒度

不要一次性做太多事。比较稳的是每轮只聚焦一种变化：

- 只整理结构
- 只改样式
- 只补资源
- 只调发布配置

这样出问题时更容易定位原因。

## 什么时候该手动保存

虽然软件有自己的状态维护，但依然建议你在这些动作后手动保存一次：

- 调整完一轮布局
- 新增页面
- 改完资源绑定
- 切到 Code 模式做过结构性修改

## 资源改动的专用节奏

资源相关建议用单独节奏，不要和布局改动混在一起：

1. 导入资源
2. 绑定资源
3. 生成资源
4. 预览验证

## 什么时候看 Release History

不是只有正式发版才看。日常也适合在下面场景打开：

- 你想确认上一次包有没有成功
- 你忘了最近一次用的是哪个 profile
- 你要回看某次失败的 log

## 什么时候看 Repository Health

建议一旦出现“今天明明没动核心逻辑，但环境像坏了”的感觉，就打开它。

## 团队协作时的建议

如果多人共用同一套工程，建议团队统一三件事：

1. 工程命名
2. 资源命名
3. Release Profile 命名

否则几周后，每个人都能打开工程，但没人知道哪个配置是正式可用的。
