# 推荐使用习惯

下面这些习惯不会让软件“多一个功能”，但会显著减少返工。

## 1. 永远先改源信息，再管生成结果

优先编辑：

- 页面 XML
- 属性
- 资源配置
- 结构树

不要把生成文件当长期手工维护入口。

## 2. 新工程先做小闭环

一个新工程创建后，建议先完成这个最小闭环：

1. 插一个控件
2. 改一个文本
3. 保存
4. Build 一次
5. 再决定是否继续铺复杂页面

## 3. 复杂页面先理结构，再细调样式

如果父子关系错了，后面很多样式调整都是在浪费时间。

## 4. 资源变更后尽快生成资源

不要累积很多资源修改后再一次性排错，这会让问题来源不清晰。

## 5. 发布前至少看三处

建议 Release 之前至少检查：

1. `Diagnostics`
2. `Debug Output`
3. `Repository Health`

## 6. 给页面和控件取可读名字

不要长期保留：

- `page1`
- `button1`
- `label2`

清晰命名会直接降低结构面板、属性面板和代码导出的理解成本。

## 7. 用 Profile 管理发布差异，不要靠记忆

只要发布参数开始分叉，就应该使用 `Release Profiles...`。

## 8. 出问题先保留现场

不要在还没确认原因时，连续做太多“也许有用”的操作。优先保留：

- 当前工程状态
- Release History
- build log
- health report

## 9. 把示例工程当参考，不要直接当生产工程

示例适合学习结构和流程，但正式项目应该尽快建立自己的目录和命名规范。

## 10. 文档和工程一起维护

只要项目开始进入多人协作，就建议把：

- SDK 绑定方式
- 发布 profile 约定
- 资源命名规则
- 目录规范

写进项目自己的协作文档里。
