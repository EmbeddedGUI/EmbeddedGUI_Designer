# EmbeddedGUI Designer UI 重规划（极简 / Android Studio 风格）

## 0. 背景与目标

你当前的问题非常明确：

- UI 逐渐“展示化”，偏离“工具化编辑器”本质；
- 信息层级不稳定，局部越改越花；
- 图标使用泛滥，文本可读性被削弱；
- 已有计划偏“零散项修复”，缺少结构性框架与持续约束。

本次重规划的目标是：

1. **回归工具属性**：优先“快读、快找、快操作”；
2. **文字优先**：有文字就不放图标；
3. **结构先于视觉**：先定义信息架构，再定义样式；
4. **低干扰高一致性**：像 Android Studio 一样克制、稳定、可预期。

---

## 1. 设计原则（必须长期执行）

### 1.1 工具优先（Tool-first）

- 不做“展示型”视觉叙事；
- 不做装饰性强调；
- 一切样式服务于定位、识别、操作效率。

### 1.2 文字优先（Text-first）

- 文本可表达时，**禁止图标替代文本**；
- 图标只允许用于：
  - 折叠/展开状态（caret）
  - 严重告警状态（error/warning）
  - 非语言化高频语义（如运行状态点）

### 1.3 层级克制（Low hierarchy noise）

- 字号层级控制在 3 级以内；
- 颜色强调控制在 2 级（默认 / 选中）；
- 去除渐变、过大圆角、重阴影。

### 1.4 一致性优先于“局部美化”

- 同类组件在所有面板完全一致；
- 允许“朴素”，不允许“每块都不一样”。

---

## 2. 信息架构（结构性方案）

目标布局采用经典 IDE 三栏结构（可伸缩）：

- **左侧（导航）**：Project / Resources
- **中间（主工作区）**：Canvas + Tabs
- **右侧（属性）**：Inspector / Property Editor
- **底部（状态）**：Status Bar（仅一行摘要）

### 2.1 左侧导航（Project + Resource）

- 用文字标签分区：`Project` / `Resources`
- 列表行：纯文本 + 缩进层级
- 行高：28~30 px（紧凑）
- 行内操作按钮默认隐藏，仅 hover 或选中显示文字按钮（如 `Rename`、`Delete`）
- 禁止每行放图标

### 2.2 中间工作区（Canvas）

- 只保留必要工具条，且以文字动作为主：`Select`、`Move`、`Align`...
- 主画布标题使用文本，不使用“Banner式”说明
- 选中反馈只使用描边/底色，不使用动画光效

### 2.3 右侧属性面板（Inspector）

- 分组顺序固定：`Layout` → `Appearance` → `Text` → `Behavior`
- 分组标题文本化，去卡片装饰
- 控件统一：label 在上，输入在下；避免混乱的左右混排
- 表单密度偏紧凑，减少“大片留白”

### 2.4 底部状态条（Status Bar）

- 单行：当前页面、选中对象、警告数量、操作提示
- 只显示关键信息，不重复左中右面板已有内容

---

## 3. 视觉系统约束（Design Tokens）

> 建议将以下规则固化为 token + lint/style check，防止再次发散。

### 3.1 间距系统（4pt）

- 仅允许：4 / 8 / 12 / 16 / 20 / 24
- 默认面板内边距：12
- 组间距：8 或 12

### 3.2 字体层级

- `title`：14（半粗）
- `body`：13（常规）
- `meta`：12（常规）
- 禁止无必要的 h1/h2 式大标题

### 3.3 颜色语义

- 背景：2 层（工作区底色 / 面板底色）
- 文字：3 层（主文本 / 次文本 / 禁用）
- 状态：仅 `selected`、`warning`、`error`
- 禁止渐变背景

### 3.4 形状与反馈

- 圆角：统一小圆角（如 4 或 6）
- 阴影：默认无阴影，仅浮层允许极轻阴影
- 动效：仅 hover / focus / selected 的基础反馈，时长 80~120ms

---

## 4. 图标策略（强约束）

### 4.1 默认规则

- **有文字就不用图标**（硬性规则）
- Tab 标题不加图标
- 按钮不加图标（除非“仅图标按钮”在空间受限且已配 tooltip）

### 4.2 例外白名单

仅以下情况允许图标：

1. Tree 展开折叠 caret
2. 严重状态（错误/警告）
3. 极少数国际通用符号且可提升效率（例如播放/暂停），同时保留文字或 tooltip

---

## 5. 关键页面规范（你这个项目最该先统一的）

### 5.1 Project Dock

- 顶部只保留文字动作按钮：`New Page` / `Import` / `Sort`
- 页面树纯文本层级；去文件类型图标
- 选中行使用底色 + 左侧细竖线，不加额外图标

### 5.2 Resource Panel

- 资源分栏文字 Tab：`Images` / `Fonts` / `Text` / `Strings`
- 每个资源条目主信息一行，次信息一行（meta）
- 操作入口统一为文字按钮（`Replace`、`Rename`、`Remove`）

### 5.3 Property Panel

- 统一分组结构和字段顺序（固定，不随组件大幅漂移）
- 输入框/下拉/开关尺寸统一
- 帮助文案弱化为次文本，不做“说明块”堆叠

## 6. 实施路线（结构化改造，不再零碎修）

### Phase A：结构冻结（1 周）

目标：先把“骨架”定住。

- 冻结三区布局（左中右 + 底部状态条）
- 冻结面板信息顺序
- 冻结按钮/Tab/列表三类基础组件结构

产出：

- `UI_STRUCTURE_SPEC.md`（可选）或本文件附录中的结构草图
- 组件清单与使用边界

### Phase B：视觉收敛（1 周）

目标：统一 token，清理花哨表现。

- 全量替换渐变、重圆角、重阴影
- 统一字体、间距、行高
- 执行“图标白名单”清理

产出：

- Token 对照表（旧 -> 新）
- 视觉回归截图（主要面板）

### Phase C：交互降噪（0.5~1 周）

目标：减小视觉噪音与重复信息。

- 合并重复提示文案
- 收敛 hover/focus 动效
- 将状态信息集中到底部状态条

产出：

- 交互规范清单
- 噪音项关闭清单

### Phase D：验收与守护（持续）

目标：防止再次“越做越花”。

- 新增 UI Review Checklist（PR 必过）
- 每周 1 次 20 分钟视觉巡检
- 每月一次“风格偏移审计”

## 8. 验收标准（可打分）

每项 0~2 分，满分 10 分，>=8 分才算通过。

1. **结构一致性**：三区布局与信息顺序是否稳定
2. **文字优先达成度**：是否严格遵守“有文字不用图标”
3. **视觉克制度**：是否无渐变/重阴影/多余装饰
4. **操作效率**：核心路径（页面、资源、属性）是否更快更清晰
5. **新增功能合规性**：新增 UI 是否遵守既定规范

## 8. UI Review Checklist（建议直接用于 PR 模板）

- [ ] 是否复用了既有基础组件（按钮/Tab/列表/输入）
- [ ] 是否使用 4pt 间距系统
- [ ] 是否新增了非白名单图标（若是，需说明理由）
- [ ] 是否引入渐变、重阴影或装饰性元素
- [ ] 是否出现重复提示文案
- [ ] 是否破坏左中右信息架构
- [ ] 是否影响核心路径效率

---

## 9. 结论

这次改造的关键不是“再改一版样式”，而是：

- 先用结构把 UI 锁住；
- 再用 token 把视觉收敛；
- 最后用流程（checklist + 巡检）防止反弹。

按这份方案执行后，编辑器会更接近 Android Studio 的“克制工具感”：

- 看起来更朴素，但更专业；
- 图标更少，但更清楚；
- 没有花哨负担，长期维护成本更低。
