# UI Style Guide（EmbeddedGUI Designer）

> 目标：确保编辑器 UI 长期保持“工具型、克制、文字优先”，避免再次视觉发散。

---

## 1. 设计基调

- 产品类型：**生产力工具 / IDE 风格编辑器**
- 视觉气质：**中性、稳定、低干扰**
- 设计优先级：
  1. 可读性
  2. 可定位性
  3. 操作效率
  4. 一致性
  5. 美观性（最后考虑）

---

## 2. 核心原则

### 2.1 文字优先

- 有文字就不放图标。
- 图标不能替代明确文本语义。
- 按钮、Tab、分组标题默认均使用文字。

### 2.2 结构优先

- 先保证信息架构稳定，再做视觉样式。
- 不允许“局部优化导致全局不一致”。

### 2.3 克制反馈

- 仅保留必要交互反馈（hover/focus/selected）。
- 禁止炫目动画、渐变强调、装饰性阴影。

### 2.4 一致性第一

- 同类组件在不同面板必须一致。
- 不允许同功能出现两套交互模式。

---

## 3. 布局规范

### 3.1 全局布局

采用 IDE 经典结构：

- 左：Project / Resources
- 中：Canvas + Tabs
- 右：Inspector（属性）
- 下：Status Bar（一行）

### 3.2 间距系统（4pt）

仅允许使用：

- 4 / 8 / 12 / 16 / 20 / 24

推荐：

- 面板内边距：12
- 组件垂直间距：8
- 分组间距：12

### 3.3 密度

- 列表行高：28~30 px
- 表单控件高度：28~32 px
- 保持紧凑，不做大面积无意义留白

---

## 4. 字体与文本规范

### 4.1 字体层级（最多 3 级）

- `title`：14 / Semibold
- `body`：13 / Regular
- `meta`：12 / Regular

禁止：

- 展示型超大标题
- 同屏出现 4 级以上文字层级

### 4.2 文案风格

- 直接、功能导向、短句。
- 避免“营销语气”“情绪化语气”。
- 提示文案优先说明“下一步动作”。

示例：

- 好：`No page selected. Choose a page from Project.`
- 差：`Looks like nothing is selected yet!`

---

## 5. 颜色与形态规范

### 5.1 颜色语义

- 背景：2 层（工作区 / 面板）
- 文本：3 层（主 / 次 / 禁用）
- 状态：3 类（selected / warning / error）

禁止：

- 渐变背景
- 非语义化高饱和色块

### 5.2 圆角与阴影

- 圆角统一小尺度（4 或 6）
- 默认无阴影
- 仅弹层可使用极轻阴影

---

## 6. 图标规范（严格）

### 6.1 默认禁止项

- 按钮图标（若已包含文字）
- Tab 图标
- 列表每行图标

### 6.2 白名单（允许）

1. Tree 折叠/展开 caret
2. error/warning 状态图标
3. 空间极限下的仅图标按钮（必须有 tooltip）

### 6.3 图标引入规则

新增图标必须满足：

- 文本无法高效替代；
- 提升操作效率；
- 在 PR 说明中写明原因；
- 不破坏整体“文字优先”原则。

---

## 7. 组件规范

### 7.1 Button

- 文字按钮优先
- 高度统一（建议 28~32）
- 主次按钮仅通过颜色/边框区分，不用图标堆叠

### 7.2 Tabs

- 纯文字 Tab
- 高度统一
- 选中态只做底色/下划线/字重变化之一

### 7.3 List / Tree

- 行高统一 28~30
- 缩进表达层级，不依赖图标
- 行内动作默认隐藏，hover 或选中显示

### 7.4 Form（Inspector）

- 结构：label 在上，输入在下
- 分组顺序固定：Layout → Appearance → Text → Behavior
- 帮助信息使用 `meta` 文本，不使用大块说明卡片

### 7.5 Status Bar

- 单行聚合状态
- 不重复其他区域信息
- 语气简洁、可执行

---

## 8. 动效规范

- 允许：hover / focus / selected
- 时长：80~120ms
- 曲线：平滑但不抢眼
- 禁止：弹跳、发光、连续闪烁、复杂过渡链

---

## 9. 可访问性与可读性

- 所有关键操作有明确文本标签
- 仅图标按钮必须带 tooltip
- 键盘焦点可见
- 对比度满足基本可读要求（尤其 meta 文本）

---

## 10. UI 评审清单（PR 必看）

- [ ] 是否遵守“有文字就不用图标”
- [ ] 是否复用现有组件，而不是新造样式
- [ ] 是否使用 4pt 间距体系
- [ ] 是否引入渐变/重阴影/装饰性元素
- [ ] 是否出现新的层级或尺寸漂移
- [ ] 是否影响核心路径效率
- [ ] 是否保持左中右结构稳定

---

## 11. 违规示例（避免重蹈覆辙）

- 同一面板中混用两种按钮高度
- 同一功能在不同区域文案不一致
- 每个模块都加一套自己的图标系统
- 为“好看”新增卡片、描边、渐变而无功能收益
- 状态提示散落在多个角落，重复且冲突

---

## 12. 维护机制

- 每周一次 20 分钟视觉巡检
- 每月一次风格偏移审计
- 新增 UI 必须通过本规范 + 计划文档中的 Checklist

如规范与业务冲突：

1. 先保核心流程效率；
2. 再在不破坏一致性的前提下做最小例外；
3. 将例外记录到文档（避免重复争论）。
